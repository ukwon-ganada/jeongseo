# 보안 설정 적용 가이드 (RLS + 직원 로그인)

이 문서는 **DB 쪽에서 딱 두 가지**만 하면 됩니다. 순서대로 따라 하세요.
① 직원용 로그인 계정 1개 만들기 → ② 보안 SQL 한 번 실행하기.
(코드/화면 쪽은 이미 이 저장소에 반영돼 있어, GitHub에 올라가면 자동 배포됩니다.)

> **왜 필요한가요?**
> 지금까지는 공개된 익명 키만 있으면 누구나 `contracts`(계약서·서명·주민번호)와
> `cases`(사건 정보)를 통째로 읽고 지울 수 있었습니다. 아래 설정을 하면
> **로그인한 직원만** 데이터에 접근하고, 외부 의뢰인은 **자기 서명 링크 한 건**만
> 열 수 있게 됩니다.

---

## ① 직원 로그인 계정 만들기 (공용 계정 1개)

1. Supabase 대시보드 접속 → 왼쪽 메뉴 **Authentication** 클릭.
2. **Providers**(또는 Sign In / Providers)에서 **Email**이 켜져 있는지 확인.
   - 켜져 있으면 그대로 두면 됩니다.
3. **Authentication → Users** 로 이동 → 오른쪽 위 **Add user → Create new user** 클릭.
4. 아래처럼 입력:
   - **Email**: 직원들이 공유할 주소 (예: `staff@jeongseo.local` — 실제 받는 메일이 아니어도 됩니다)
   - **Password**: **길고 강한 비밀번호** (최소 12자, 예: 문구+숫자+기호). 이게 사실상 앱 열쇠입니다.
   - **Auto Confirm User**: **반드시 체크** ✅ (안 하면 로그인 시 "이메일 인증" 때문에 막힙니다)
5. **Create user** 클릭.
6. 이 **이메일 + 비밀번호**를 직원들에게 공유하세요. (앱 로그인 화면에 그대로 입력)

> 나중에 비밀번호를 바꾸려면 같은 화면에서 사용자를 열어 **Reset password / Update** 하면 됩니다.

---

## ② 보안 SQL 실행하기 (RLS 켜기 + 서명 링크 안전화)

1. Supabase 대시보드 → 왼쪽 메뉴 **SQL Editor** 클릭 → **New query**.
2. 이 저장소의 파일
   **`supabase/migrations/20260704_rls_and_sign.sql`** 을 열어
   **내용 전체를 복사**해서 붙여넣습니다.
3. 오른쪽 아래 **Run**(▶) 클릭.
4. 아래쪽에 `Success. No rows returned` 비슷한 메시지가 나오면 완료입니다.
   (여러 번 실행해도 안전하게 만들어 두었습니다.)

이 SQL이 하는 일:
- `contracts`, `cases` 테이블에 **RLS(행 보안)** 를 켜서 익명 접근을 막습니다.
- **로그인한 직원**에게만 목록/저장/삭제 권한을 줍니다.
- 외부 서명 링크는 테이블을 직접 열지 않고 **토큰을 서버에서 검증하는 함수**
  (`sign_fetch`, `sign_submit`)로만 그 한 건에 접근하게 합니다.
- **서명 무결성 지문(SHA-256)** 을 서버가 직접 계산해 저장 → 위조가 불가능해집니다.

---

## ③ 잘 됐는지 확인

**SQL Editor**에서 아래를 실행:

```sql
-- 두 테이블 모두 t(true) 여야 정상
select relname, relrowsecurity
from pg_class
where relname in ('contracts','cases');
```

그리고 실제 동작 확인:

- ✅ **직원**: 앱을 새로고침하면 **로그인 화면**이 뜨고, ①에서 만든 계정으로
  로그인하면 계약서 목록·작성이 평소처럼 됩니다.
- ✅ **외부 의뢰인**: 기존처럼 `...?sign=토큰` 링크를 열면 **로그인 없이** 계약서가
  보이고 서명 제출이 됩니다.
- ✅ **차단 확인(선택)**: 로그아웃 상태(또는 시크릿 창)에서 앱을 열면 로그인 화면에
  막혀 목록이 보이지 않아야 합니다.

## 자주 겪는 문제

- **로그인이 계속 실패해요**
  → 계정 만들 때 **Auto Confirm User** 체크를 안 했을 수 있습니다.
    Users에서 해당 사용자를 열어 이메일을 **Confirm** 처리하거나, 사용자를 지우고
    체크한 채로 다시 만드세요.

- **서명 링크가 "유효하지 않은 링크"라고 떠요**
  → ② SQL이 실행되지 않았을 수 있습니다. SQL Editor에서 다시 Run 하세요.
    또는 `pgcrypto` 확장 오류가 났다면 맨 위
    `create extension if not exists pgcrypto;` 줄만 따로 한 번 실행한 뒤 전체를 다시 Run.

- **되돌리고 싶어요(임시)**
  → 아래로 RLS만 잠깐 끌 수 있습니다. **단, 끄면 다시 전체 노출 상태가 됩니다.**
    ```sql
    alter table public.contracts disable row level security;
    alter table public.cases     disable row level security;
    ```

## 참고: 공개돼도 되는 것 / 안 되는 것

- **익명 키(anon key)** 는 원래 브라우저에 공개되는 값입니다. RLS를 켠 지금은
  이 키만으로는 아무 데이터도 못 읽으므로 그대로 두어도 안전합니다.
- **service_role 키**(있다면)는 절대 프론트엔드/깃허브에 넣지 마세요. 서버 전용입니다.
- 직원 **비밀번호**가 이제 진짜 열쇠입니다. 안전하게 공유·보관하세요.
